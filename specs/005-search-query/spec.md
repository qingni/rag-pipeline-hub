# Feature Specification: 搜索查询功能

**Feature Branch**: `005-search-query`  
**Created**: 2025-12-25  
**Status**: Draft  
**Input**: User description: "搜索查询功能，依赖已实现的向量索引功能"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 语义搜索查询 (Priority: P1)

用户在搜索框中输入自然语言问题或关键词，系统将查询文本转换为向量，通过已构建的向量索引进行相似度检索，返回最相关的文档片段列表。这是RAG系统的核心检索能力。

**Why this priority**: 这是搜索查询模块的核心功能，是用户与知识库交互的主要入口，直接决定了RAG系统的可用性。

**Independent Test**: 可以通过输入一个问题（如"什么是向量数据库？"），验证系统是否返回相关的文档片段，并检查返回结果的相关性评分。

**Acceptance Scenarios**:

1. **Given** 用户已有构建好的向量索引（包含1000个文档片段），**When** 用户输入查询"如何使用Python连接数据库"，**Then** 系统在2秒内返回Top10相关文档片段，每个结果包含文本内容、相似度分数和来源信息
2. **Given** 用户输入查询文本，**When** 查询文本为空或仅包含空格，**Then** 系统提示"请输入有效的查询内容"
3. **Given** 用户输入查询，**When** 索引中没有相似度高于阈值的结果，**Then** 系统返回空列表并提示"未找到相关内容，请尝试其他关键词"

---

### User Story 2 - 搜索结果展示与交互 (Priority: P1)

用户在前端界面查看搜索结果，结果以卡片列表形式展示，每个卡片显示文档片段的核心内容、相似度分数、来源文档名称。用户可以点击查看完整内容或跳转到原文档。

**Why this priority**: 良好的结果展示直接影响用户体验，是搜索功能可用性的关键。

**Independent Test**: 可以通过执行搜索后验证结果卡片是否正确显示所有必要信息，点击操作是否正常响应。

**Acceptance Scenarios**:

1. **Given** 搜索返回了5条结果，**When** 结果页面加载完成，**Then** 显示5张结果卡片，每张包含：文本摘要（最多200字）、相似度分数（百分比形式）、来源文档名、文档片段位置
2. **Given** 用户点击某个结果卡片的"查看详情"，**When** 详情弹窗打开，**Then** 显示完整的文档片段内容、元数据信息和相关上下文
3. **Given** 搜索结果超过10条，**When** 用户滚动到底部，**Then** 自动加载更多结果（分页加载）

---

### User Story 3 - 搜索配置与过滤 (Priority: P2)

用户可以配置搜索参数，包括选择目标索引、设置返回结果数量（TopK）、调整相似度阈值、选择相似度计算方法。高级用户可以通过这些配置优化搜索效果。

**Why this priority**: 配置功能增强了搜索的灵活性，但基础搜索功能可以使用默认配置运行。

**Independent Test**: 可以通过修改配置参数后执行搜索，验证结果是否符合配置的预期。

**Acceptance Scenarios**:

1. **Given** 用户打开搜索配置面板，**When** 面板显示，**Then** 展示以下可配置项：目标索引（下拉选择）、返回数量（1-100）、相似度阈值（0-1）、相似度方法（余弦/欧氏/点积）
2. **Given** 用户设置TopK=5且相似度阈值=0.7，**When** 执行搜索，**Then** 最多返回5条结果，且每条结果的相似度分数≥0.7
3. **Given** 系统存在多个向量索引，**When** 用户选择特定索引进行搜索，**Then** 只在该索引范围内检索

---

### User Story 4 - 搜索历史记录 (Priority: P2)

系统记录用户的搜索历史，用户可以查看历史搜索记录、快速重复执行历史查询、清除历史记录。这提升了用户的使用效率和体验。

**Why this priority**: 历史记录是提升用户体验的增强功能，核心搜索功能不依赖它。

**Independent Test**: 可以通过执行多次搜索后验证历史记录是否正确保存，点击历史记录是否能重新执行搜索。

**Acceptance Scenarios**:

1. **Given** 用户执行了一次搜索，**When** 搜索完成，**Then** 该查询自动保存到搜索历史，包含查询文本、时间戳、结果数量
2. **Given** 用户打开搜索历史面板，**When** 点击某条历史记录，**Then** 自动填充查询框并执行搜索
3. **Given** 用户点击"清除历史"按钮，**When** 确认操作，**Then** 所有搜索历史被清除

---

### User Story 5 - 多索引联合搜索 (Priority: P3)

用户可以选择多个向量索引进行联合搜索，系统合并各索引的搜索结果并按相似度排序返回。这支持跨知识库的统一检索场景。

**Why this priority**: 这是高级功能，适用于多知识库场景，单索引搜索已能满足基本需求。

**Independent Test**: 可以通过选择多个索引执行搜索，验证结果是否来自所有选中的索引，排序是否正确。

**Acceptance Scenarios**:

1. **Given** 用户选择了"技术文档"和"产品手册"两个索引，**When** 执行搜索，**Then** 返回结果包含来自两个索引的文档，按相似度统一排序
2. **Given** 联合搜索返回结果，**When** 查看结果列表，**Then** 每条结果明确标注来源索引名称
3. **Given** 用户设置TopK=10进行联合搜索，**When** 两个索引各有足够结果，**Then** 返回合并后的Top10，而非每个索引各返回10条

---

### Edge Cases

- 当查询文本过长（超过模型最大输入长度）时，系统如何处理？
- 当选择的向量索引不存在或已被删除时，系统如何响应？
- 当Embedding服务不可用时，搜索功能如何降级处理？
- 当搜索请求超时时，系统如何通知用户？
- 当用户快速连续发起多次搜索时，如何处理并发请求？
- 当相似度阈值设置过高导致无结果时，是否给出调整建议？

## Requirements *(mandatory)*

### Functional Requirements

#### 前端界面需求

- **FR-UI-001**: 搜索页面顶部提供搜索输入框，支持回车键或点击按钮触发搜索
- **FR-UI-002**: 搜索结果以卡片列表形式展示，支持分页或无限滚动加载
- **FR-UI-003**: 每个结果卡片显示：文本摘要、相似度分数（百分比）、来源文档名、操作按钮（查看详情）
- **FR-UI-004**: 提供搜索配置面板，包含索引选择、TopK设置、相似度阈值、相似度方法选择
- **FR-UI-005**: 提供搜索历史侧边栏，显示最近的搜索记录（最多50条）
- **FR-UI-006**: 搜索过程中显示加载状态，搜索完成后显示结果数量和耗时
- **FR-UI-007**: 界面样式与现有模块（文档处理、向量化、向量索引）保持一致

#### 核心搜索功能

- **FR-001**: 系统必须支持接收自然语言查询文本，调用Embedding服务转换为查询向量
- **FR-002**: 系统必须调用向量索引服务执行相似度检索，返回TopK最相关结果
- **FR-003**: 系统必须支持配置返回结果数量（TopK），范围1-100，默认值10
- **FR-004**: 系统必须支持配置相似度阈值，范围0-1，默认值0.5
- **FR-005**: 系统必须支持选择相似度计算方法：余弦相似度（默认）、欧氏距离、点积
- **FR-006**: 系统必须支持选择单个或多个目标向量索引进行搜索
- **FR-007**: 搜索结果必须包含：文档片段ID、文本内容、相似度分数、来源文档信息、元数据

#### 结果处理

- **FR-008**: 系统必须对搜索结果按相似度分数降序排列
- **FR-009**: 多索引搜索时，系统必须合并各索引结果并统一排序
- **FR-010**: 系统必须支持结果分页，每页默认10条，支持自定义每页数量
- **FR-011**: 系统必须对返回的文本内容进行摘要处理，默认最多显示200字符

#### 历史记录

- **FR-012**: 系统必须自动记录每次搜索的查询文本、时间戳、配置参数、结果数量
- **FR-013**: 系统必须支持查看搜索历史列表，按时间倒序排列
- **FR-014**: 系统必须支持点击历史记录快速重新执行搜索
- **FR-015**: 系统必须支持清除全部或单条搜索历史

#### 错误处理

- **FR-016**: 当查询文本为空时，系统必须返回明确的错误提示
- **FR-017**: 当Embedding服务调用失败时，系统必须返回友好的错误信息并记录日志
- **FR-018**: 当向量索引不可用时，系统必须返回明确的错误状态
- **FR-019**: 搜索请求必须设置超时限制（默认30秒），超时后返回错误提示

### Key Entities

- **SearchQuery（搜索查询）**: 代表一次搜索请求，包含查询文本、目标索引列表、TopK、相似度阈值、相似度方法等参数
- **SearchResult（搜索结果）**: 代表单条搜索结果，包含文档片段ID、文本内容、相似度分数、来源索引、来源文档、元数据
- **SearchHistory（搜索历史）**: 代表一条搜索历史记录，包含查询ID、查询文本、搜索配置、结果数量、执行时间、创建时间
- **SearchConfig（搜索配置）**: 代表搜索的配置参数，包含默认索引、默认TopK、默认阈值、默认相似度方法

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户输入查询到看到搜索结果的端到端响应时间在3秒以内（95%分位）
- **SC-002**: 搜索结果的相关性：用户对Top5结果的满意度达到80%以上（通过用户反馈或点击率衡量）
- **SC-003**: 系统支持至少20个并发搜索请求，总体吞吐量达到10 QPS
- **SC-004**: 搜索功能可用性达到99.5%，Embedding服务故障时有明确的降级提示
- **SC-005**: 用户可在5秒内完成搜索配置的修改（索引选择、参数调整）
- **SC-006**: 搜索历史记录准确率100%，每次搜索都被正确记录
- **SC-007**: 多索引联合搜索的结果合并正确率100%，排序符合相似度降序

## Assumptions

- 假设向量索引模块已完成并可用，提供标准的相似度检索API
- 假设Embedding服务已完成并可用，提供文本到向量的转换API
- 假设单次查询文本长度不超过2000字符
- 假设搜索历史存储在本地数据库，不跨设备同步
- 假设默认使用余弦相似度作为相似度计算方法
- 假设前端使用与现有模块一致的技术栈（Vue 3 + TDesign）

## Dependencies

- **向量索引模块（004-vector-index）**: 提供向量相似度检索能力
- **向量Embedding模块（003-vector-embedding）**: 提供查询文本的向量化能力
- **文档处理模块（001-document-processing）**: 提供文档元数据信息
- **文档分块模块（002-doc-chunking）**: 提供文档片段的原始内容

## Out of Scope

- 关键词搜索（BM25等传统检索方法）
- 混合检索（向量+关键词融合）
- 搜索结果的重排序（Reranking）
- 搜索结果的个性化推荐
- 搜索分析和统计报表
- 搜索结果的导出功能
- 多语言查询支持（当前仅支持中文和英文）
