# Feature Specification: 文档分块功能

**Feature Branch**: `002-doc-chunking`  
**Created**: 2025-12-05  
**Status**: Draft  
**Input**: User description: "帮我新建个分支，完成文档分块功能，文档分块要使用文档加载输出结果"

## Clarifications

### Session 2025-12-05

- Q: 当用户同时运行多个分块任务时（例如对不同文档或同一文档使用不同策略），系统应该如何处理并发？ → A: 支持并发但有队列限制（如最多3个并发任务，其他排队等待）
- Q: 对于"按语义分块"策略，当文档中语义边界不明显或算法无法识别合理的语义分界点时，系统应该如何降级处理？ → A: 自动降级为按段落分块策略，并在结果中标注降级原因
- Q: 分块结果的数据结构应该如何组织以便于后续向量化和检索？ → A: 返回包含文档级元信息+chunks列表的字典；每个chunk包含content+metadata（页码、词数、块编号等）
- Q: 当分块处理过程中发生错误（如超时、资源不足），系统是否应该保留部分成功的分块结果？ → A: 保留部分成功的分块结果，并在结果中标记为"部分完成"状态，记录错误发生的位置
- Q: 对于分块历史列表的存储和查询，当历史记录数量增长到数百条时，系统应该如何优化性能和用户体验？ → A: 分页加载（如每页20条），支持按文档名、策略、时间等维度过滤和排序
- Q: 对于"按标题分块"策略，当文档没有明确的标题结构时，系统应该如何处理？ → A: 检测到无标题结构时，提示用户并建议切换到其他策略（按段落/按字数）

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 基于加载结果的文档分块 (Priority: P1)

用户需要对已完成加载和解析的文档进行智能分块处理，将文档内容切分成合适大小的块，以便后续进行向量嵌入和语义检索。用户可以选择不同的分块策略，预览分块结果，并保存分块数据供下游模块使用。

**Why this priority**: 这是连接文档解析和向量嵌入的关键桥梁，是实现智能检索的基础步骤。没有合理的文档分块，后续的向量嵌入和检索质量将大打折扣。

**Independent Test**: 可以通过加载已保存的文档解析JSON结果，选择分块策略（如按段落、按字数），配置分块参数（如块大小、重叠度），查看分块预览，并验证生成的分块JSON文件来完全测试此功能。

**Acceptance Scenarios**:

1. **Given** 用户在文档分块页面，**When** 用户从下拉列表中选择一个已解析的文档，**Then** 系统加载该文档的解析结果并显示文档基本信息（文档名、页数、字符数）
2. **Given** 文档解析结果已加载，**When** 用户选择"按段落分块"策略并设置块大小为500字符、重叠度为50字符，**Then** 系统实时显示预计生成的块数量和分块参数配置
3. **Given** 分块参数已配置，**When** 用户点击"开始分块"按钮，**Then** 系统执行分块处理并显示处理进度条
4. **Given** 分块处理完成，**When** 用户查看分块结果，**Then** 系统在右侧内容区域显示所有分块的列表视图，包含块索引、块内容预览（前100字符）、字符数、词数等统计信息
5. **Given** 分块结果已显示，**When** 用户点击某个分块条目，**Then** 系统展开显示该块的完整内容和元数据（块位置、父文档引用、分块策略等）
6. **Given** 分块处理完成，**When** 系统保存分块结果，**Then** 系统生成JSON文件（格式：文档名_chunk_时间戳.json）保存到results/chunking目录，并在数据库中建立索引

---

### User Story 2 - 多种分块策略支持 (Priority: P2)

用户需要根据不同的文档类型和使用场景，选择最适合的分块策略，以优化后续的检索和生成效果。系统支持多种分块策略，每种策略适用于不同的文档结构和内容特征。

**Why this priority**: 不同类型的文档（如技术手册、小说、研究论文）需要不同的分块策略才能保证内容的完整性和检索的准确性。这增强了系统的适用性和灵活性。

**Independent Test**: 可以通过使用同一文档分别应用不同的分块策略（按段落、按字数、按语义、按标题），对比各策略的分块结果，验证每种策略的分块逻辑是否符合预期。

**Acceptance Scenarios**:

1. **Given** 用户选择"按字数分块"策略，**When** 用户设置块大小为1000字符、重叠度为100字符，**Then** 系统按固定字符数切分文档，保证块之间有100字符的重叠
2. **Given** 用户选择"按段落分块"策略，**When** 用户设置最大块大小为800字符，**Then** 系统以自然段落为基本单位分块，当段落超过800字符时进一步切分
3. **Given** 用户选择"按标题分块"策略，**When** 文档包含结构化标题（H1、H2、H3等），**Then** 系统按标题层级分块，每个块包含标题及其下的所有内容直到下一个同级或上级标题；如果文档不包含可识别的标题结构，系统提示用户"未检测到标题结构"并建议切换到按段落或按字数分块策略
4. **Given** 用户选择"按语义分块"策略，**When** 用户设置最小块大小为300字符、最大块大小为1200字符，**Then** 系统使用语义相似度算法识别语义边界，在保持语义完整性的前提下进行分块；如果无法识别明显的语义边界，系统自动降级为按段落分块并在结果中标注降级原因

---

### User Story 3 - 分块结果管理和导出 (Priority: P3)

用户需要管理已生成的分块结果，包括查看历史分块记录、对比不同分块策略的结果、删除不需要的分块数据、导出分块结果供外部使用。

**Why this priority**: 分块是一个迭代优化的过程，用户可能需要尝试多种分块策略并对比效果。良好的结果管理功能提升用户体验和工作效率。

**Independent Test**: 可以通过查看分块历史列表，选择多个分块结果进行对比，删除测试用的分块数据，导出分块JSON文件并验证格式完整性来独立测试。

**Acceptance Scenarios**:

1. **Given** 用户在分块管理页面，**When** 用户查看分块历史列表，**Then** 系统显示所有历史分块记录，包含文档名、分块策略、块数量、创建时间、文件大小等信息
2. **Given** 用户选择两个不同策略的分块结果，**When** 用户点击"对比"按钮，**Then** 系统并排显示两个分块结果的统计对比（块数量、平均块大小、最大/最小块大小、重叠情况）
3. **Given** 用户选择一个分块结果，**When** 用户点击"删除"按钮并确认，**Then** 系统删除该分块的JSON文件和数据库记录，并更新分块历史列表
4. **Given** 用户选择一个分块结果，**When** 用户点击"导出"按钮，**Then** 系统提供JSON格式和CSV格式的导出选项，下载包含所有分块数据的文件

---

### Edge Cases

- 当用户选择的文档解析结果文件不存在或已损坏时，系统应提示错误并建议重新加载文档
- 当文档内容过短（如少于100字符）导致无法按指定参数分块时，系统应提示调整分块参数或将整个文档作为单一块
- 当用户设置的块大小超过文档总长度时，系统应生成单一块并提示用户
- 当用户设置的重叠度大于或等于块大小时，系统应提示参数冲突并要求调整
- 当分块处理过程中系统资源不足或处理超时时，系统应保存已处理的部分并提示用户
- 当分块过程因错误中断时，系统必须保留已成功分块的部分结果，在结果状态中标记为"部分完成（partial）"，并记录错误发生的位置（如"已完成前80%，第1500个字符处失败"）
- 当文档包含大量特殊字符或非文本内容时，系统应正确处理并在统计信息中标注
- 当使用按标题分块策略但文档不包含标题结构时，系统应提示用户并建议使用其他策略，避免执行无效操作
- 当多个用户同时对同一文档进行分块时，系统应为每次分块生成唯一的时间戳标识避免文件冲突

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须能够读取并解析results/load或results/parse目录下的文档加载/解析JSON结果文件
- **FR-002**: 系统必须提供文档选择界面，显示所有可用的已解析文档列表，包含文档名、处理时间、文件大小等信息
- **FR-003**: 系统必须支持至少四种分块策略：按字数分块、按段落分块、按标题分块、按语义分块
- **FR-003a**: 当按语义分块策略无法识别明显的语义边界时，系统必须自动降级为按段落分块策略，并在分块结果的元数据中记录降级原因和受影响的块数量
- **FR-003b**: 当按标题分块策略检测到文档不包含可识别的标题结构时，系统必须中止分块并提示用户"未检测到标题结构，建议使用按段落或按字数分块策略"
- **FR-004**: 用户必须能够配置分块参数，包括块大小（最小值50字符、最大值5000字符）、重叠度（0-500字符）
- **FR-005**: 系统必须在用户调整参数时实时显示预计生成的块数量和平均块大小
- **FR-006**: 系统必须在分块处理过程中显示进度条和当前处理状态
- **FR-006a**: 系统必须支持最多3个并发分块任务，超出的任务自动进入队列等待，用户可查看队列状态和任务顺序
- **FR-007**: 系统必须在右侧内容区域以列表形式展示所有分块结果，包含块索引、内容预览、字符数、词数等统计信息
- **FR-008**: 用户必须能够点击任意分块条目查看该块的完整内容和元数据
- **FR-009**: 系统必须将分块结果保存为JSON格式文件到results/chunking目录，文件命名格式为：文档名_chunk_时间戳.json
- **FR-010**: 分块结果JSON文件必须采用以下数据结构：
  - **文档级元信息**：document_id、document_name、source_file、total_chunks、chunking_strategy、chunking_params（块大小、重叠度等）、status（completed/partial）、created_at、processing_time、error_info（如适用，包含错误位置和原因）
  - **chunks列表**：每个chunk包含：
    - **content**: 块的文本内容
    - **metadata**: chunk_id、chunk_index、char_count、word_count、page_number（如有）、start_position、end_position、包含的标题（如有）、段落数、降级标记（如适用）
- **FR-011**: 系统必须在数据库中为每个分块结果建立索引，记录文档ID、分块策略、块数量、创建时间、文件路径等信息
- **FR-012**: 系统必须提供分块历史管理界面，显示所有历史分块记录并支持查询、对比、删除操作
- **FR-012a**: 分块历史列表必须支持分页加载（默认每页20条记录），并提供按文档名、分块策略、创建时间等维度的过滤和排序功能
- **FR-013**: 系统必须支持分块结果的导出功能，提供JSON和CSV两种格式
  - **JSON格式**：完整保留文档级元信息和chunks列表的嵌套结构
  - **CSV格式**：扁平化导出，列包括：chunk_id、chunk_index、document_name、strategy、char_count、word_count、page_number、chunk_content（换行符转义）
- **FR-014**: 系统必须对分块参数进行验证，当参数不合理时（如重叠度大于块大小）显示错误提示并阻止执行
- **FR-015**: 系统必须在左侧控制面板提供文档选择、策略选择、参数配置、操作按钮等控件，在右侧内容区域显示分块结果和统计信息

### Key Entities

- **ChunkingTask**: 表示分块任务，包含源文档引用、分块策略、分块参数、任务状态（pending/running/completed/partial/failed）、创建时间等属性
- **ChunkingStrategy**: 表示分块策略，包含策略名称、策略类型（字数/段落/标题/语义）、策略描述、默认参数等属性
- **DocumentChunk**: 表示单个文档块，包含content（块文本内容）和metadata（chunk_id、chunk_index、char_count、word_count、page_number、start_position、end_position、包含的标题、段落数、降级标记等）
- **ChunkingResult**: 表示分块结果，包含文档级元信息（document_id、document_name、source_file、total_chunks、chunking_strategy、chunking_params、status、created_at、processing_time、error_info）和chunks列表，以及统计信息（平均块大小、最大/最小块大小）、JSON文件路径

### Assumptions

- 文档加载和解析结果已经保存为标准JSON格式，包含完整的文本内容和文档结构信息
- 分块功能作为独立模块运行，不依赖其他模块的实时状态，通过读取已保存的JSON文件获取输入
- 分块结果JSON文件保存在results/chunking目录，文件名格式为：文档名_chunk_时间戳.json
- 按语义分块策略使用基础的句子边界检测和语义相似度算法，不依赖复杂的NLP模型（复杂模型可在后续版本中优化）
- 系统默认采用按段落分块策略，块大小默认为500字符，重叠度默认为50字符
- 分块处理在后端同步执行，单个文档的分块时间预期在10秒内完成
- 系统支持的文档大小与文档加载模块一致（最大50MB），对应的最大字符数约为5000万字符

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户能够在30秒内完成文档选择、策略配置、启动分块的完整流程
- **SC-002**: 对于标准大小的文档（约10000字符），分块处理时间在5秒内完成
- **SC-003**: 不同分块策略的分块结果准确率达到95%以上（验证方法：随机选择10个文档，每个文档抽取10个块，共100个样本；评分标准：块边界在语义完整单元内得1分，跨语义单元得0分）
- **SC-004**: 系统能够稳定处理包含50000字符的大型文档，分块时间在30秒内完成
- **SC-005**: 90%的用户在首次使用时能够理解不同分块策略的区别并成功完成分块任务（测试方法：10名未接触过系统的用户进行首次使用测试，任务成功率≥9/10）
- **SC-006**: 分块结果JSON文件的数据完整性和格式正确性达到100%
- **SC-007**: 系统对不合理参数的验证准确率达到100%，防止用户配置错误导致的处理失败（测试方法：执行参数验证测试套件，覆盖所有边界条件和无效输入组合）
- **SC-008**: 分块结果的管理操作（查看、对比、删除、导出）的响应时间在2秒内完成
- **SC-009**: 分块历史列表在包含500条记录时，分页加载和过滤操作的响应时间在1秒内完成
