## Clarifications

### Session 2026-02-02

- Q: 多模态内容（图片、表格）的向量化策略？ → A: 图片优先使用 base64 直接向量化，失败时降级为文本描述；表格使用 Markdown 格式文本向量化
- Q: 批量向量化的并发控制策略？ → A: 支持配置最大并发数，默认 5 个并发请求，支持速率限制
- Q: 向量化进度的实时反馈机制？ → A: 支持 WebSocket/SSE 实时推送进度，前端展示进度条和已完成块数
- Q: 增量向量化的触发条件？ → A: 当分块结果有更新时，仅对新增/变更的块进行向量化，跳过未变化的块
- Q: 向量化结果的缓存策略？ → A: 基于内容哈希的缓存机制，相同内容不重复计算向量
- Q: 向量化失败的重试和降级策略？ → A: 支持指数退避重试，多模型降级（主模型失败时切换备用模型）
- Q: 文档特征提取维度（用于智能模型推荐）？ → A: 基于内容特征：主要语言（中/英/多语言）、内容领域（技术/商业/法律）、多模态内容比例
- Q: 批量文档的模型推荐策略？ → A: 统一推荐 + 异常标注：分析所有文档综合特征后提供统一推荐，同时标注特征差异较大的文档供用户单独处理
- Q: 模型推荐结果的展示方式？ → A: 推荐卡片：展示推荐模型、推荐理由、特征匹配度可视化（如雷达图/进度条展示各维度匹配情况）
- Q: 嵌入模型能力特征数据来源？ → A: 静态配置 + 管理界面：预置默认配置，同时提供管理界面让用户调整模型能力评分
- Q: 推荐算法的匹配策略？ → A: 加权评分算法：各维度（语言、领域、多模态）分别计算匹配度得分，按可配置权重加权求和，取最高分模型

## Background

本优化基于现有 `003-vector-embedding` 分支的向量化功能，结合 `002-doc-chunking-opt` 的多模态分块成果，旨在通过以下方面提升向量化模块的效率和效果：

### 现有向量化功能分析

当前系统支持：
- **单文本向量化**：对单个文本进行向量化（仅后端 API）
- **批量文本向量化**：对多个文本批量向量化（仅后端 API）
- **文档分块向量化**：基于分块结果进行向量化
- **多模型支持**：BGE-M3、Qwen3-Embedding-8B、Hunyuan-Embedding、Jina-Embeddings-v4

### 优化方向

1. **多模态向量化**：支持图片 base64 直接向量化（使用多模态模型）、表格结构化向量化
2. **批量处理优化**：并发控制、批量大小自适应、失败重试机制
3. **增量向量化**：基于内容变更检测，避免重复计算
4. **进度反馈优化**：实时进度推送、详细状态展示
5. **向量缓存机制**：基于内容哈希的缓存，提升重复内容处理效率
6. **前端体验优化**：进度可视化、批量操作、结果对比
7. **智能模型推荐**：基于文档内容特征（语言、领域、多模态比例）自动推荐最适合的嵌入模型
8. **批量文档智能推荐**：批量添加文档时，分析综合特征提供统一推荐，并标注特征差异较大的文档
9. **推荐结果可视化**：通过推荐卡片展示模型推荐结果，包含推荐理由和特征匹配度可视化（雷达图/进度条）
10. **模型能力配置管理**：预置主流模型的能力特征配置，同时提供管理界面让用户调整模型在各维度的能力评分
11. **加权评分推荐算法**：基于加权评分算法进行模型推荐，各维度（语言匹配度、领域匹配度、多模态支持度）分别计算得分，按可配置权重加权求和，推荐综合得分最高的模型

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 多模态内容向量化 (Priority: P1)

用户需要对包含图片和表格的分块结果进行向量化，系统应智能选择合适的向量化策略，以提升多模态内容的检索效果。

**Why this priority**: 多模态内容是文档中重要的信息载体，有效的向量化是实现精准检索的基础。与 002-doc-chunking-opt 的多模态分块功能紧密配合。

**Independent Test**: 可以通过对包含图片和表格的分块结果进行向量化，验证不同类型内容的向量化策略是否正确应用。

**Acceptance Scenarios**:

1. **Given** 用户选择了包含图片分块的文档，**When** 执行向量化操作，**Then** 系统优先使用图片 base64 调用多模态模型（qwen3-vl-embedding-8b）进行向量化
2. **Given** 图片向量化失败（模型不可用或图片格式不支持），**When** 系统检测到失败，**Then** 自动降级为使用图片描述文本（caption）进行向量化，并在结果中标记降级原因
3. **Given** 用户选择了包含表格分块的文档，**When** 执行向量化操作，**Then** 系统使用表格的 Markdown 格式内容进行文本向量化
4. **Given** 分块结果包含多种类型（文本、表格、图片、代码），**When** 执行向量化，**Then** 系统根据 chunk_type 自动选择对应的向量化策略

---

### User Story 2 - 批量向量化性能优化 (Priority: P1)

用户需要对大量分块进行高效向量化，系统应支持并发处理、自适应批量大小、失败重试，以提升处理效率和稳定性。

**Why this priority**: 大文档可能包含数百甚至上千个分块，高效的批量处理是生产环境的必要能力。

**Independent Test**: 可以通过对不同规模的分块结果进行向量化，验证并发控制、批量处理、重试机制是否正常工作。

**Acceptance Scenarios**:

1. **Given** 用户选择了包含 500 个分块的文档，**When** 执行向量化操作，**Then** 系统按配置的并发数（默认 5）并行处理，总处理时间相比串行处理减少 70% 以上
2. **Given** 向量化过程中某个批次请求失败（网络超时），**When** 检测到失败，**Then** 系统自动进行指数退避重试（最多 3 次），成功后继续处理
3. **Given** 单个批次重试 3 次后仍然失败，**When** 重试耗尽，**Then** 系统记录失败分块信息，继续处理剩余分块，最终返回部分成功结果
4. **Given** 用户配置了自定义并发数和批量大小，**When** 执行向量化，**Then** 系统按用户配置执行，并在超出合理范围时给出警告

---

### User Story 3 - 向量化进度实时反馈 (Priority: P1)

用户需要在向量化过程中实时看到处理进度，包括已完成数量、预估剩余时间、当前状态等信息。

**Why this priority**: 对于大文档向量化，用户需要了解处理进度以判断是否正常执行，这是基本的用户体验需求。

**Independent Test**: 可以通过触发向量化操作，验证前端是否实时更新进度信息，进度数据是否准确。

**Acceptance Scenarios**:

1. **Given** 用户开始向量化操作（100 个分块），**When** 处理进行中，**Then** 前端进度条实时更新，显示"已完成 X/100 (X%)"
2. **Given** 向量化正在执行，**When** 前端每 500ms 更新一次状态，**Then** 显示当前处理速度（如"25 块/秒"）和预估剩余时间
3. **Given** 向量化过程中发生部分失败，**When** 有分块处理失败，**Then** 前端显示失败数量和失败原因摘要，但进度条继续更新
4. **Given** 用户在向量化过程中刷新页面，**When** 页面重新加载，**Then** 系统能够恢复显示当前进度状态（任务未完成时）

---

### User Story 4 - 增量向量化 (Priority: P2)

用户需要在文档重新分块后，仅对新增或变更的分块进行向量化，避免对未变化内容重复计算。

**Why this priority**: 减少不必要的计算可以节省时间和 API 成本，尤其对于大型文档的频繁更新场景。

**Independent Test**: 可以通过对同一文档进行两次分块和向量化，验证第二次是否正确跳过未变化的分块。

**Acceptance Scenarios**:

1. **Given** 文档已完成向量化（100 个分块），**When** 文档更新并重新分块后有 10 个新增分块、5 个修改分块，**Then** 增量向量化仅处理这 15 个分块，跳过其余 85 个未变化分块
2. **Given** 系统检测到分块内容变更，**When** 执行增量向量化，**Then** 前端显示"检测到 15 个变更分块，跳过 85 个未变化分块"
3. **Given** 用户选择"强制全量向量化"，**When** 执行操作，**Then** 系统忽略缓存，对所有分块重新计算向量
4. **Given** 增量向量化完成，**When** 查看向量化结果，**Then** 结果中包含所有分块的向量（包括复用的和新计算的），并标注每个向量的来源（缓存/新计算）

---

### User Story 5 - 向量缓存机制 (Priority: P2)

用户需要系统对相同内容的向量化结果进行缓存，避免重复调用 API，提升效率并降低成本。

**Why this priority**: 相同内容在不同文档或不同时间可能被多次向量化，缓存可以显著提升效率。

**Independent Test**: 可以通过对包含重复内容的分块进行向量化，验证第二次是否命中缓存。

**Acceptance Scenarios**:

1. **Given** 文档 A 有一个分块内容与文档 B 的某分块完全相同，**When** 对文档 B 进行向量化，**Then** 该分块直接从缓存获取向量，不调用 API
2. **Given** 用户查看向量化结果，**When** 有分块命中缓存，**Then** 结果中标记"来自缓存"，并显示缓存命中率统计
3. **Given** 用户执行"清除向量缓存"操作，**When** 操作完成，**Then** 后续向量化全部重新计算，不使用缓存
4. **Given** 缓存达到配置的最大容量，**When** 新内容需要缓存，**Then** 系统按 LRU 策略淘汰旧缓存

---

### User Story 6 - 多模型切换与对比 (Priority: P2)

用户需要能够方便地切换向量化模型，并对比不同模型的向量化结果，以选择最适合的模型。

**Why this priority**: 不同模型在不同场景下表现不同，提供对比功能帮助用户做出最佳选择。

**Independent Test**: 可以通过使用不同模型对同一文档进行向量化，验证对比功能是否正确展示差异。

**Acceptance Scenarios**:

1. **Given** 用户选择了一个已向量化的文档，**When** 用户更换模型并点击"重新向量化"，**Then** 系统使用新模型生成新的向量化结果，保留历史结果
2. **Given** 文档有多个使用不同模型的向量化结果，**When** 用户点击"模型对比"，**Then** 系统展示各模型的处理时间、向量维度、成功率等对比信息
3. **Given** 用户在对比视图中，**When** 用户选择某个模型的结果，**Then** 该结果被设为"活跃"状态，用于后续检索

---

### User Story 7 - 向量化结果可视化 (Priority: P3)

用户需要直观地查看向量化结果，包括向量维度分布、处理状态分布、模型使用情况等统计信息。

**Why this priority**: 可视化帮助用户理解向量化结果的质量和特征，便于后续优化。

**Independent Test**: 可以通过完成向量化后查看结果页面，验证统计图表是否正确展示。

**Acceptance Scenarios**:

1. **Given** 向量化完成，**When** 用户查看结果页面，**Then** 显示处理状态饼图（成功/失败/跳过）、处理时间分布直方图
2. **Given** 文档包含多种类型分块，**When** 查看统计信息，**Then** 显示各类型分块的向量化成功率（文本 X%、表格 Y%、图片 Z%）
1. **Given** 用户查看向量列表，**When** 鼠标悬停在某个向量上，**Then** 显示该向量的详细信息（维度、处理时间、来源、内容预览）

---

### User Story 8 - 智能嵌入模型推荐 (Priority: P1)

用户需要系统根据文档内容特征自动推荐最适合的嵌入模型，减少用户的选择负担，提升向量化质量。

**Why this priority**: 模型选择直接影响向量化和检索效果，智能推荐是优化用户体验和检索质量的关键功能。

**Independent Test**: 可以通过上传不同类型文档（中文技术文档、英文商业文档、多模态文档），验证系统推荐结果是否合理。

**Acceptance Scenarios**:

1. **Given** 用户上传了一份中文技术文档，**When** 系统分析文档特征后，**Then** 展示推荐卡片，显示推荐模型（如 BGE-M3）、推荐理由（"检测到中文技术内容，该模型在中文技术领域表现优异"）、以及特征匹配度雷达图
2. **Given** 用户上传了包含大量图片的多模态文档，**When** 系统分析到多模态比例超过 30%，**Then** 优先推荐支持多模态的嵌入模型（如 qwen3-vl-embedding-8b）
3. **Given** 系统完成模型推荐，**When** 用户查看推荐卡片，**Then** 可以看到 Top 3 推荐模型及其各维度（语言、领域、多模态）的匹配度得分
4. **Given** 用户不满意推荐结果，**When** 用户选择"手动选择模型"，**Then** 系统展示所有可用模型列表供用户选择

### User Story 9 - 批量文档智能推荐 (Priority: P1)

用户需要批量添加多个文档时，系统能够分析所有文档的综合特征并提供统一的模型推荐，同时标注异常文档。

**Why this priority**: 批量处理是常见场景，统一推荐简化用户决策，异常标注保证处理质量。

**Independent Test**: 可以通过批量上传特征相似和特征差异较大的文档组合，验证推荐和异常标注功能。

**Acceptance Scenarios**:

1. **Given** 用户批量上传 10 个中文技术文档，**When** 系统分析综合特征后，**Then** 提供统一的模型推荐，推荐卡片显示"综合分析 10 个文档：主要语言-中文 100%，领域-技术 90%，多模态比例 15%"
2. **Given** 批量文档中有 2 个英文法律文档（与其他中文技术文档特征差异较大），**When** 系统检测到特征偏离超过阈值，**Then** 在推荐卡片中标注这 2 个异常文档，并建议"2 个文档特征差异较大，建议单独处理"
3. **Given** 用户查看异常文档列表，**When** 用户点击某个异常文档，**Then** 系统展示该文档的独立推荐结果，供用户决定是否单独处理
4. **Given** 用户选择"统一处理所有文档"，**When** 执行向量化，**Then** 系统使用统一推荐的模型处理所有文档，并在结果中标注异常文档的处理效果评估

### User Story 10 - 模型能力配置管理 (Priority: P2)

管理员需要能够查看和调整系统预置的模型能力特征配置，以适应实际使用情况和新模型的添加。

**Why this priority**: 模型能力会随版本更新变化，管理界面允许灵活调整，确保推荐准确性。

**Independent Test**: 可以通过管理界面修改某模型的能力评分，验证推荐结果是否相应变化。

**Acceptance Scenarios**:

1. **Given** 管理员进入模型配置管理页面，**When** 页面加载完成，**Then** 展示所有预置模型的能力特征表格，包含各维度评分（语言支持、领域专长、多模态能力等）
2. **Given** 管理员修改某模型的"中文支持度"评分从 80 调整为 95，**When** 保存配置后，**Then** 后续中文文档的模型推荐结果会相应调整
3. **Given** 管理员点击"添加新模型"，**When** 填写模型名称和各维度能力评分后保存，**Then** 新模型出现在可选模型列表和推荐候选中
4. **Given** 管理员需要恢复默认配置，**When** 点击"重置为默认"，**Then** 所有模型能力评分恢复为系统预置值

### Edge Cases
- 当所有向量化请求都失败时，系统应返回完整的错误信息和失败原因统计
- 当图片格式不支持（如 SVG）时，系统应自动降级为文本描述向量化
- 当多模态模型不可用时，系统应降级为纯文本模型处理，跳过图片 base64 向量化
- 当缓存服务不可用时，系统应正常执行向量化（不使用缓存），并记录警告日志
- 当并发数设置过高导致 API 限流时，系统应自动降低并发数并重试
- 当内容哈希冲突时（极低概率），系统应检测向量维度不匹配并重新计算
- 当进度推送连接断开时，前端应自动重连并恢复进度显示
- 当向量化任务被用户取消时，系统应停止处理并保存已完成的结果
- 当文档内容过短无法准确分析语言/领域特征时，系统应返回低置信度推荐并提示用户手动选择
- 当批量文档中所有文档特征差异都较大（无法提取共性）时，系统应提示用户分组处理或逐文档推荐
- 当所有模型能力评分数据不可用时，系统应退回到默认模型选择，不提供推荐
- 当推荐算法计算超时时，系统应返回基于简化规则的快速推荐结果

## Requirements *(mandatory)*

### Functional Requirements

#### 多模态向量化

- **FR-001**: 系统必须支持图片分块的多模态向量化，优先使用 image_base64 字段调用多模态模型（qwen3-vl-embedding-8b）
- **FR-002**: 系统必须实现图片向量化的降级机制：多模态失败时降级为 caption 文本向量化，并记录降级原因
- **FR-003**: 系统必须支持表格分块的向量化，使用 table_markdown 字段的 Markdown 格式内容
- **FR-004**: 系统必须根据 chunk_type 字段自动选择向量化策略（text→文本模型，image→多模态模型，table→文本模型，code→文本模型）
- **FR-005**: 系统必须支持多模态模型配置，可选模型：qwen3-vl-embedding-8b（支持图文）、其他多模态模型（扩展）

#### 批量处理优化

- **FR-006**: 系统必须支持可配置的并发控制，默认并发数为 5，范围 1-20
- **FR-007**: 系统必须支持可配置的批量大小，默认 50，范围 10-200
- **FR-008**: 系统必须实现指数退避重试机制：初始延迟 1s，最大延迟 32s，抖动 ±25%，最多重试 3 次
- **FR-009**: 系统必须支持部分成功返回：批次失败时继续处理剩余批次，最终返回成功/失败分块明细
- **FR-010**: 系统必须在检测到 API 限流时自动降低并发数，并在限流解除后逐步恢复

#### 增量向量化

- **FR-011**: 系统必须支持基于内容哈希的增量向量化检测，识别新增/修改/未变化的分块
- **FR-012**: 系统必须为每个分块计算内容哈希（SHA-256），用于变更检测
- **FR-013**: 增量向量化必须跳过内容哈希未变化且已有向量的分块，直接复用历史向量
- **FR-014**: 系统必须支持"强制全量向量化"选项，忽略增量检测，重新计算所有向量
- **FR-015**: 增量向量化结果必须标注每个向量的来源：cached（复用）、computed（新计算）

#### 向量缓存

- **FR-016**: 系统必须实现基于内容哈希的向量缓存机制，缓存键为：hash(content + model_name)
- **FR-017**: 系统必须支持配置缓存容量（默认 10000 条）和过期时间（默认 7 天）
- **FR-018**: 系统必须实现 LRU 缓存淘汰策略，容量满时淘汰最久未使用的缓存
- **FR-019**: 系统必须支持手动清除缓存操作（按文档/按模型/全部清除）
- **FR-020**: 缓存命中时必须验证向量维度与当前模型匹配，不匹配时重新计算

#### 进度反馈

- **FR-021**: 系统必须支持向量化进度的实时推送，使用 WebSocket 或 SSE 协议
- **FR-022**: 进度信息必须包含：总分块数、已完成数、失败数、当前批次、处理速度、预估剩余时间
- **FR-023**: 系统必须支持任务状态查询 API，返回任务的完整进度信息
- **FR-024**: 前端必须实现进度条组件，实时展示向量化进度和状态
- **FR-025**: 系统必须支持向量化任务的取消操作，取消后保存已完成的结果

#### 结果管理与对比

- **FR-026**: 系统必须保存每次向量化的完整结果记录，包括模型、参数、统计信息
- **FR-027**: 系统必须支持同一文档多个向量化结果的保存和管理
- **FR-028**: 系统必须支持向量化结果对比功能，展示不同模型/参数的结果差异
- **FR-029**: 系统必须支持设置"活跃"向量化结果，用于后续检索操作
- **FR-030**: 系统必须支持向量化结果的删除操作（非活跃结果可删除）

#### 前端交互优化

- **FR-031**: 向量化页面必须显示文档的分块类型分布（文本/表格/图片/代码块数量）
- **FR-032**: 向量化配置界面必须支持高级选项（并发数、批量大小、增量模式、缓存选项）
- **FR-033**: 向量化结果页面必须展示统计图表（成功率、类型分布、处理时间分布）
- **FR-034**: 前端必须支持向量化结果的导出功能（JSON 格式）
- **FR-035**: 前端必须在向量化完成后显示缓存命中率统计

#### 智能模型推荐

- **FR-036**: 系统必须支持基于文档内容特征（语言、领域、多模态比例）分析文档并提取特征向量
- **FR-037**: 系统必须预置主流嵌入模型的能力特征配置，包括语言支持度、领域专长、多模态能力等维度评分
- **FR-038**: 系统必须提供管理界面允许用户查看和调整模型在各维度的能力评分
- **FR-039**: 系统必须实现加权评分算法进行模型推荐：各维度（语言匹配度、领域匹配度、多模态支持度）分别计算得分，按可配置权重加权求和
- **FR-040**: 系统必须支持配置推荐算法的各维度权重，默认权重为：语言 40%、领域 35%、多模态 25%
- **FR-041**: 批量添加文档时，系统必须分析所有文档的综合特征后提供统一的模型推荐
- **FR-042**: 批量推荐时，系统必须标注特征差异较大的文档（偏离综合特征超过阈值，默认阈值为偏离度 > 0.3，可通过配置调整），供用户单独处理
- **FR-043**: 推荐结果必须以推荐卡片形式展示，包含：推荐模型名称、推荐理由摘要、特征匹配度可视化（雷达图/进度条）
- **FR-044**: 推荐卡片必须展示 Top 3 推荐模型，用户可点击查看详细对比信息
- **FR-045**: 系统必须支持用户忽略推荐结果，手动选择任意可用模型

### Non-Functional Requirements

- **NFR-001**: 批量向量化性能：100 个分块的向量化应在 30 秒内完成（5 并发，网络正常）
- **NFR-002**: 进度推送延迟：进度更新应在 500ms 内推送到前端
- **NFR-003**: 缓存查询性能：缓存命中查询应在 10ms 内返回
- **NFR-004**: 内存使用：向量缓存内存占用不超过 500MB（10000 条 4096 维向量）
- **NFR-005**: 增量检测性能：1000 个分块的变更检测应在 1 秒内完成
- **NFR-006**: API 响应时间：向量化状态查询 API 应在 100ms 内返回

### Assumptions

- 多模态模型（qwen3-vl-embedding-8b）已部署并可通过 API 访问
- 分块结果来自 002-doc-chunking-opt，包含 chunk_type 和多模态元数据（image_base64、table_markdown）
- WebSocket/SSE 连接稳定，网络断开时前端能够自动重连
- 缓存存储使用本地内存或 Redis（可配置），本规格默认使用本地内存
- API 限流由提供商控制，系统通过错误响应检测并自动调整
- 内容哈希使用 SHA-256 算法，冲突概率可忽略

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 多模态内容向量化成功率达到 95% 以上（图片向量化含降级策略）
- **SC-002**: 批量向量化性能提升：500 分块处理时间相比串行减少 70% 以上
- **SC-003**: 进度推送延迟 p95 < 500ms
- **SC-004**: 增量向量化正确率 100%：准确识别所有变更/未变化的分块
- **SC-005**: 缓存命中场景下向量化速度提升 90% 以上
- **SC-006**: 向量化失败时自动重试成功率达到 80% 以上（针对临时性错误）
- **SC-007**: 前端进度可视化准确率 100%：进度数据与实际处理状态一致
- **SC-008**: 用户能够在 30 秒内完成模型切换和重新向量化操作
- **SC-009**: 智能模型推荐准确率达到 85% 以上（推荐模型与用户最终选择一致的比例）
- **SC-010**: 文档特征分析延迟 p95 < 2s（单文档）、p95 < 5s（10 个文档批量）
- **SC-011**: 批量推荐的异常文档识别准确率达到 90% 以上（真正需要单独处理的文档被正确标注）
- **SC-012**: 推荐卡片展示完整性 100%：包含推荐模型、理由、匹配度可视化
