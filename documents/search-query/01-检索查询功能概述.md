## 1. 功能概述

### 1.1 核心功能
检索查询模块是 RAG (Retrieval-Augmented Generation) 系统中负责执行语义检索的核心组件。它基于向量索引模块构建的索引结构，提供多种检索方式，包括向量搜索、混合检索（稠密+稀疏双路召回）、多 Collection 联合搜索等能力，为下游的生成模块提供高质量的上下文信息。

### 1.2 主要特性
- ✅ **向量搜索**: 基于稠密向量的相似度搜索，支持 Top-K 和阈值过滤
- ✅ **混合检索**: 稠密+稀疏双路召回 → RRF 粗排 → Reranker 精排
- ✅ **BM25 稀疏向量查询**: 基于 jieba 分词的 BM25 稀疏向量生成查询向量
- ✅ **Reranker 精排**: 使用 qwen3-reranker-4b 模型（远程 API）对候选结果精排，支持 Task Instruction 前缀
- ✅ **查询增强 (Query Enhancement)**: 🆕 通过 LLM 一次调用同时完成 Query Rewrite（查询改写）和 Multi-query（多查询变体生成），提升检索召回质量
- ✅ **三层防御体系**: 🆕 候选配额控制 → 近重复内容去重 → 动态阈值过滤 + 文档来源多样性控制，系统性保障召回质量
- ✅ **多 Collection 联合搜索**: 跨知识库搜索，支持跨模型联合检索（不同 Collection 可绑定不同 Embedding 模型）
- ✅ **分页支持**: 🆕 支持 page/page_size 分页参数
- ✅ **智能降级**: 稀疏向量不可用时自动降级为纯稠密检索，Reranker 不可用时跳过精排，查询增强失败时降级使用原始查询
- ✅ **查询历史**: 完整的查询历史记录与统计
- ✅ **召回调试日志**: 🆕 专用的 `search_recall_debug.log` 文件，记录每次查询的粗排/精排/最终召回详情

### 1.3 设计原则
1. **混合检索优先**: 默认启用稠密+稀疏双路召回，提升关键词和语义的综合召回能力
2. **优雅降级**: 任何组件不可用时自动降级，保证检索可用性
3. **多层质量保障**: 三层防御体系 + Reranker 动态阈值，系统性过滤低质量结果
4. **可观测性**: 完整的查询耗时（含查询增强、嵌入、BM25、检索、Reranker 各阶段）、检索模式、Reranker 状态等指标追踪

### 2.4 与文本生成模块的关系

检索查询模块不仅服务独立的搜索页面，也直接支撑文本生成页的自动上下文检索。当前文本生成功能的典型调用链路如下：

```text
Generation.vue
  -> generationStore.loadAvailableCollections()
     -> GET /api/v1/search/collections
  -> generationStore.retrieveContext(question)
     -> POST /api/v1/search/hybrid
  -> 将 results[] 映射为 generation ContextItem[]
  -> POST /api/v1/generation/stream
```

文本生成页当前主要消费混合检索结果中的以下字段：

- `text_content` -> 生成上下文的 `content`
- `source_document` -> 生成上下文的 `source_file`
- `similarity_score` -> 生成上下文的 `similarity`
- `chunk_id` -> 生成上下文的 `chunk_id`
- `metadata` -> 生成上下文的 `metadata`

## 5. 工作流程

### 5.1 语义搜索流程

```
1. 用户输入查询文本
   └─> 调用 Embedding 服务向量化

2. 执行语义搜索
   └─> POST /api/v1/search
       ├─> search_service.search()
       ├─> 验证查询文本（非空、长度限制）
       ├─> 获取目标索引列表
       ├─> 调用 Embedding 服务向量化
       ├─> 在各索引中搜索（milvus_provider.search）
       ├─> 合并排序结果
       ├─> 应用阈值过滤
       └─> 记录查询历史

3. 返回搜索结果
   └─> 查询ID、结果列表、耗时
```

### 5.2 混合检索流程（含查询增强 + 三层防御）

```
1. 用户输入查询文本
   └─> POST /api/v1/search/hybrid

2. 查询增强（Query Enhancement）  🆕
   ├─> QueryEnhancementService.enhance_query()
   ├─> LLM 一次调用 → JSON 结构化输出
   │   ├─> Query Rewrite（必选）：改写查询使语义更清晰
   │   └─> Multi-query 判断（可选）：复杂查询生成 2-3 个变体
   ├─> 改写后的 rewritten_query 替换原始查询
   └─> 失败时降级使用原始查询（不影响后续流程）

3. Embedding 向量化
   ├─> 按 Collection 绑定的 Embedding 模型分组
   ├─> 各模型并行嵌入查询文本
   └─> Multi-query: 并行嵌入所有变体查询

4. 检索执行
   ├─> 单 Collection:
   │   ├─> Step 0.5: BM25 自动生成 query sparse 向量
   │   ├─> Step 1: 稠密+稀疏双路召回 + RRF 粗排融合
   │   └─> Multi-query: 并行执行变体查询检索 → 合并去重
   └─> 多 Collection（≤5个）:
       ├─> asyncio.gather 并行在各 Collection 检索
       ├─> 第 1 层防御：每文档候选配额控制（MAX_CHUNKS_PER_DOC）
       └─> 合并候选集

5. 三层防御体系  🆕
   ├─> 第 1 层：每文档候选配额控制（MAX_CHUNKS_PER_DOC=10）
   │   └─> 限制同一文档在候选集中的 chunk 数量
   ├─> 第 1.5 层：近重复内容去重（NEAR_DUPLICATE_THRESHOLD=0.5）
   │   ├─> Layer 1: 精确哈希去重（跨文档完全相同内容）
   │   └─> Layer 2: N-gram Jaccard 近重复去重（字符级相似度）
   ├─> Reranker 精排（qwen3-reranker-4b, 全量打分不截取）
   ├─> 第 2 层：动态阈值过滤
   │   ├─> 静态阈值: RERANKER_SCORE_THRESHOLD=0.4
   │   ├─> 动态阈值: Top1 × DYNAMIC_RATIO=0.6, cap=0.5
   │   └─> 保底召回: RERANKER_MIN_RESULTS=2
   └─> 第 3 层：文档来源多样性控制（MAX_RESULTS_PER_DOC=2）

6. 返回结果
   ├─> top_k 截取 → 分页（page/page_size）
   ├─> 保存搜索历史
   └─> 附加 query_enhancement 信息

7. 降级策略
   ├─> 查询增强失败 → 降级使用原始查询
   ├─> 稀疏向量为空/无效 → 降级到纯稠密检索 (dense_only)
   └─> Reranker 不可用 → 跳过精排，直接返回 RRF 粗排结果
```

### 5.3 多 Collection 联合搜索流程

```
1. 用户选择多个 Collection（最多5个）+ 输入查询

2. 跨模型嵌入  🆕
   ├─> 按 Embedding 模型分组 Collection
   └─> 各模型并行嵌入查询文本（确保各 Collection 在正确语义空间检索）

3. 并行检索
   └─> asyncio.gather() 并行在各 Collection 执行检索
       ├─> 每个 Collection 独立执行 hybrid/dense_only 检索
       ├─> 第 1 层防御：每 Collection 候选配额控制
       └─> Multi-query: 变体查询的额外并行检索

4. 合并 + 三层防御
   ├─> 合并所有 Collection 候选集
   ├─> 第 1.5 层防御：近重复内容去重
   ├─> 统一 Reranker 精排（Reranker 基于文本，天然支持跨模型）
   ├─> 第 2 层防御：动态阈值过滤
   └─> 第 3 层防御：文档来源多样性控制

5. 返回 Top-K 结果
```

### 5.4 文本生成页自动检索流程

```text
1. 用户在文本生成页输入问题
   -> Generation.vue / GenerationInput.vue

2. 前端加载可用知识库
   -> GET /api/v1/search/collections

3. 用户选择一个或多个 Collection
   -> selectedCollectionIds

4. 发起混合检索
   -> POST /api/v1/search/hybrid
      参数通常包含:
      - query_text
      - collection_ids
      - top_k
      - threshold=0.3

5. 前端将检索结果映射为生成上下文
   -> text_content / source_document / similarity_score / chunk_id / metadata

6. 生成模块携带 context 调用 /api/v1/generation/generate 或 /stream
```

## 6. 关键特性

### 6.1 查询增强（Query Enhancement）🆕

```
原始查询
  │
  ├─> QueryEnhancementService.enhance_query()
  │   ├─> LLM 一次调用（qwen3.5-35b-a3b, temperature=0.3）
  │   ├─> System Prompt + User Prompt → JSON 输出
  │   │
  │   └─> 输出:
  │       ├─> rewritten_query: "改写后的主查询"
  │       ├─> is_complex: true/false
  │       └─> sub_queries: ["变体1", "变体2"]（仅 is_complex=true）
  │
  ├─> 主查询路径: rewritten_query → Embedding → 检索
  └─> Multi-query 路径（is_complex=true）:
      ├─> 所有变体并行 Embedding
      ├─> 各变体并行检索
      ├─> 合并去重（按 vector_id，保留最高分）
      └─> 统一送入 Reranker 精排
```

**设计参考**:
- LangChain MultiQueryRetriever: LLM 生成多查询变体，合并去重
- LlamaIndex QueryTransformEngine: 查询改写 + 子问题分解
- Cohere search_queries_only: 一次调用生成多个搜索查询
- Google Vertex AI Search: 自动查询改写和扩展

### 6.2 三层防御体系 🆕

```
候选集（RRF 粗排后 / 多 Collection 合并后）
  │
  ├─> 第 1 层：每文档候选配额控制 (MAX_CHUNKS_PER_DOC=10)
  │   └─> 限制同一文档在候选集中最多占 N 条，防止单一文档垄断
  │
  ├─> 第 1.5 层：近重复内容去重 (NEAR_DUPLICATE_THRESHOLD=0.5)
  │   ├─> Layer 1: 精确哈希去重（O(n)）
  │   │   └─> 跨文档完全相同的 chunk 文本 → 保留第一个
  │   └─> Layer 2: N-gram Jaccard 近重复去重（O(n²)）
  │       └─> 字符级 4-gram Jaccard 相似度 ≥ 阈值 → 跨文档的近似重复内容
  │
  ├─> Reranker 精排（全量打分，不在此处截取 top_k）
  │
  ├─> 第 2 层：Reranker 动态阈值过滤
  │   ├─> 静态阈值: 0.4（绝对下限，参考 Dify/Cohere 实践）
  │   ├─> 动态阈值: Top1_score × 0.6，上限 cap=0.5（参考 Pinecone 实践）
  │   ├─> 实际阈值: max(静态, 动态)
  │   └─> 保底召回: 至少保留 2 条（参考 Dify/LangChain 保底策略）
  │
  └─> 第 3 层：文档来源多样性控制 (MAX_RESULTS_PER_DOC=2)
      └─> 限制每个文档来源在最终结果中最多占 N 个位置
```

### 6.3 混合检索架构

```
查询文本
  │
  ├─> 查询增强 → rewritten_query + sub_queries
  │
  ├─> Embedding 模型 → 稠密查询向量 (dense_vector)
  ├─> BM25 Service → 稀疏查询向量 (sparse_vector)
  │
  ├─> Milvus hybrid_search()
  │   ├─> AnnSearchRequest(稠密, embedding 字段)
  │   ├─> AnnSearchRequest(稀疏, sparse_embedding 字段)
  │   └─> RRFRanker(k=60) 融合排序 → 候选集 (top_n=20)
  │
  ├─> 三层防御体系（候选配额→近重复去重→动态阈值→多样性控制）
  │
  ├─> Reranker 精排 (qwen3-reranker-4b, 远程 API)
  │   ├─> Task Instruction 前缀（Instruction-tuned 模型）
  │   └─> query + candidate_text → reranker_score（全量打分）
  │
  └─> 最终结果 (top_k + 分页)
```

### 6.4 BM25 稀疏向量查询

- 查询时调用 `BM25SparseService.encode_query()` 生成查询稀疏向量
- 加载索引构建阶段持久化的 BM25 统计数据: `data/bm25_stats/{collection_name}_bm25_stats.json`
- 使用 **jieba** 中文分词器对查询文本进行分词

### 6.5 降级策略

| 条件 | 行为 | 影响 |
|------|------|------|
| 查询增强 LLM 调用失败 | 🆕 降级使用原始查询 | 失去查询改写和多查询能力 |
| 查询增强 JSON 解析失败 | 🆕 降级使用原始查询 | 同上 |
| 稀疏向量为空/无效 | 降级到纯稠密检索 (dense_only) | 失去关键词召回能力 |
| Collection 无 sparse_embedding 字段 | 降级到纯稠密检索 | 同上 |
| Reranker 不可用 | 跳过精排，直接返回 RRF 粗排结果 | 排序精度下降 |
| Multi-query 跨模型场景 | 🆕 降级为仅使用主查询（变体跳过） | 减少嵌入开销 |

### 6.6 Reranker 精排

- **模型**: qwen3-reranker-4b（远程 OpenAI-compatible API）
- **输入**: query (含 Task Instruction 前缀) + candidate_text 对
- **输出**: reranker_score（0~1），全量打分不截取
- **Task Instruction**: 🆕 `Instruct: {task_description}\nQuery: {actual_query}` 格式，提升排序准确度
- **健康检查**: `GET /api/v1/search/reranker/health`
- **降级**: 不可用时自动跳过，返回 RRF 粗排结果

### 6.7 跨模型联合搜索 🆕

不同 Collection 可绑定不同的 Embedding 模型，跨模型联合搜索时：
- 按 Embedding 模型分组 Collection
- 各模型并行嵌入查询文本（确保各 Collection 在正确语义空间检索）
- Reranker 精排基于文本，天然支持跨模型精排

参考业界最佳实践：
- LlamaIndex MultiIndex: 各 Index 绑定自己的 QueryEngine，独立 embed + 检索
- LangChain EnsembleRetriever: 各 retriever 独立运行，RRF/Reranker 融合结果
