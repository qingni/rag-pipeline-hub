## 1. 功能概述

### 1.1 核心功能
检索查询模块是 RAG (Retrieval-Augmented Generation) 系统中负责执行语义检索的核心组件。它基于向量索引模块构建的索引结构，提供多种检索方式，包括向量搜索、混合检索（稠密+稀疏双路召回）、多 Collection 联合搜索等能力，为下游的生成模块提供高质量的上下文信息。

### 1.2 主要特性
- ✅ **向量搜索**: 基于稠密向量的相似度搜索，支持 Top-K 和阈值过滤
- ✅ **混合检索**: 稠密+稀疏双路召回 → RRF 粗排 → Reranker 精排
- ✅ **BM25 稀疏向量查询**: 基于 jieba 分词的 BM25 稀疏向量生成查询向量
- ✅ **Reranker 精排**: 使用 bge-reranker-v2-m3 模型对候选结果精排
- ✅ **多 Collection 联合搜索**: 跨知识库搜索，支持 score/round_robin 合并策略
- ✅ **批量搜索**: 支持批量查询向量同时搜索
- ✅ **智能降级**: 稀疏向量不可用时自动降级为纯稠密检索，Reranker 不可用时跳过精排
- ✅ **查询历史**: 完整的查询历史记录与统计

### 1.3 设计原则
1. **混合检索优先**: 默认启用稠密+稀疏双路召回，提升关键词和语义的综合召回能力
2. **优雅降级**: 任何组件不可用时自动降级，保证检索可用性
3. **可观测性**: 完整的查询耗时、检索模式、Reranker 状态等指标追踪

### 2.2 模块职责

| 模块 | 职责 | 主要文件 |
|------|------|----------|
| **前端UI** | 检索交互、结果展示 | `HybridSearch.vue`, `VectorSearch.vue` |
| **API层** | 检索接口暴露（5个端点） | `api/vector_index.py`（检索部分） |
| **检索服务** | 检索核心逻辑 | `vector_index_service.py`（检索方法） |
| **BM25 查询** | 稀疏向量查询生成 | `bm25_service.py`（encode_query） |
| **Reranker** | 精排重排序 | `reranker_service.py` |
| **Provider层** | 向量数据库搜索操作 | `milvus_provider.py`（search 方法） |

### 2.3 与向量索引模块的关系

```
┌──────────────────┐         ┌──────────────────┐
│   向量索引模块     │ ──────→ │   检索查询模块     │
│  (Index/Write)   │         │  (Search/Read)   │
├──────────────────┤         ├──────────────────┤
│ • 创建/管理索引    │         │ • 向量搜索        │
│ • Collection 管理  │         │ • 混合检索        │
│ • BM25 稀疏向量构建│         │ • 联合搜索        │
│ • 推荐引擎        │         │ • Reranker 精排   │
│ • 操作日志        │         │ • 查询历史        │
└──────────────────┘         └──────────────────┘
```

> **说明**: 检索查询模块在代码层面与向量索引模块共享同一组服务文件（如 `vector_index_service.py`），但在职责划分上，检索查询聚焦于"读取/查询"操作，而向量索引聚焦于"写入/管理"操作。

## 5. 工作流程

### 5.1 向量搜索流程

```
1. 用户输入查询文本
   └─> 调用 Embedding 服务向量化

2. 执行向量搜索
   └─> POST /vector-index/indexes/{id}/search
       ├─> vector_index_service.search()
       ├─> 验证索引状态
       ├─> 验证向量维度
       ├─> 调用 Provider 搜索（milvus_provider.search）
       ├─> 应用阈值过滤
       └─> 记录查询历史

3. 返回搜索结果
   └─> 向量ID、相似度分数、元数据
```

### 5.2 混合检索流程

```
1. 用户输入查询文本
   └─> 调用 Embedding 服务获取稠密查询向量

2. 执行混合检索
   └─> POST /vector-index/hybrid-search
       ├─> Step 0.5: BM25 自动生成 query sparse 向量
       │   └─> bm25_service.encode_query(query_text)
       ├─> Step 1: 稠密+稀疏双路召回 + RRF 粗排融合
       │   ├─> AnnSearchRequest（稠密, embedding 字段）
       │   ├─> AnnSearchRequest（稀疏, sparse_embedding 字段）
       │   └─> RRFRanker 融合排序（k=60）
       ├─> Step 2: Reranker 精排（bge-reranker-v2-m3）
       │   └─> 不可用时自动降级，跳过精排
       └─> Step 3: 返回最终结果

3. 降级策略
   ├─> 稀疏向量为空/无效 → 降级到纯稠密检索
   └─> Reranker 不可用 → 跳过精排，直接返回 RRF 粗排结果
```

### 5.3 多 Collection 联合搜索流程

```
1. 用户选择多个 Collection + 输入查询
   └─> 调用 Embedding 服务向量化

2. 执行联合搜索
   └─> POST /vector-index/multi-search
       ├─> 并行在每个 Collection 中执行搜索
       ├─> 根据合并策略聚合结果
       │   ├─> score: 按相似度分数全局排序
       │   └─> round_robin: 轮询交替选取
       └─> 返回 Top-K 结果

3. 返回聚合后的搜索结果
```

## 6. 关键特性

### 6.1 混合检索架构

```
查询文本
  │
  ├─> Embedding 模型 → 稠密查询向量 (dense_vector)
  ├─> BM25 Service → 稀疏查询向量 (sparse_vector)
  │
  ├─> Milvus hybrid_search()
  │   ├─> AnnSearchRequest(稠密, embedding 字段)
  │   ├─> AnnSearchRequest(稀疏, sparse_embedding 字段)
  │   └─> RRFRanker(k=60) 融合排序 → 候选集 (top_n=20)
  │
  ├─> Reranker 精排 (bge-reranker-v2-m3)
  │   └─> query + candidate_text → reranker_score
  │
  └─> 最终结果 (top_k=5)
```

### 6.2 BM25 稀疏向量查询

- 查询时调用 `BM25SparseService.encode_query()` 生成查询稀疏向量
- 加载索引构建阶段持久化的 BM25 统计数据: `data/bm25_stats/{collection_name}_bm25_stats.json`
- 使用 **jieba** 中文分词器对查询文本进行分词

### 6.3 降级策略

| 条件 | 行为 | 影响 |
|------|------|------|
| 稀疏向量为空/无效 | 降级到纯稠密检索 (dense_only) | 失去关键词召回能力 |
| Collection 无 sparse_embedding 字段 | 降级到纯稠密检索 | 同上 |
| Reranker 不可用 | 跳过精排，直接返回 RRF 粗排结果 | 排序精度下降 |

### 6.4 Reranker 精排

- **模型**: bge-reranker-v2-m3
- **输入**: query + candidate_text 对
- **输出**: reranker_score（0~1）
- **健康检查**: `GET /vector-index/reranker/health`
- **降级**: 不可用时自动跳过，返回 RRF 粗排结果

### 6.5 多 Collection 联合搜索

| 合并策略 | 说明 | 适用场景 |
|---------|------|---------|
| score | 按相似度分数全局排序 | 关注最相关结果 |
| round_robin | 轮询交替从各 Collection 选取 | 均衡各知识库覆盖 |

## 7. 性能指标

### 7.1 搜索性能

| 向量数量 | Top-K | FLAT | HNSW |
|---------|-------|------|------|
| 10,000 | 10 | ~5ms | ~1ms |
| 100,000 | 10 | ~50ms | ~2ms |
| 1,000,000 | 10 | ~500ms | ~5ms |

### 7.2 混合检索性能（参考）

| 阶段 | 典型耗时 |
|------|---------|
| BM25 稀疏向量生成 | ~5ms |
| 双路召回 + RRF 融合 | ~80ms |
| Reranker 精排 | ~45ms |
| 总耗时 | ~130ms |

### 7.3 支持能力

| 指标 | 能力 |
|------|------|
| 最大 Top-K | 100 |
| 批量搜索 | 支持 |
| 阈值过滤 | 支持 |
| 混合检索 | 支持（稠密+稀疏双路） |
| 联合搜索 | 支持（跨 Collection） |
| 精排 | 支持（bge-reranker-v2-m3） |

## 8. 后续优化方向

1. **多路召回**: 支持更多召回策略（如 TF-IDF、关键词匹配等）
2. **查询理解**: 查询改写、意图识别、关键词提取
3. **检索评估**: 引入 NDCG、MRR 等指标自动评估检索质量
4. **缓存机制**: 热点查询缓存，降低重复查询延迟
5. **流式返回**: 支持检索结果流式输出
