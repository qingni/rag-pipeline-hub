# 向量数据库迁移方案

> 本文档记录从 JSON 文件存储迁移到专业向量数据库的技术方案，用于支持大规模场景下的向量检索需求。

## 1. 背景与动机

### 1.1 现有方案

当前项目采用 **JSON 文件 + SQLite 数据库** 的分层存储架构：

| 存储位置 | 存储内容 |
|---------|---------|
| SQLite 数据库 | 任务级别元信息（document_id, chunking_result_id, 模型信息等） |
| JSON 文件 | 向量数据 + 分块元数据 + 原始文本 |

### 1.2 迁移动机

| 场景 | JSON 文件方案 | 向量数据库方案 |
|-----|--------------|---------------|
| 数据规模 | 适合中小规模（万级） | 支持大规模（百万/亿级） |
| 检索性能 | 线性扫描，随数据量增长而下降 | ANN 索引，亚秒级响应 |
| 混合查询 | 需要手动实现 | 原生支持向量检索 + 元数据过滤 |
| 横向扩展 | 不支持 | 支持分布式部署 |
| 实时更新 | 需重写文件 | 支持增量更新 |

### 1.3 迁移范围说明

> ⚠️ **重要提示**：迁移到向量数据库 **只涉及向量化结果**，而非所有 JSON 文件。

#### 数据处理流程

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   文档加载    │ ──▶ │   文档分块    │ ──▶ │   向量化     │
│  (解析原文)   │     │  (切分文本)   │     │ (生成embedding)│
└──────────────┘     └──────────────┘     └──────────────┘
       │                    │                    │
       ▼                    ▼                    ▼
   原始文档JSON          分块结果JSON          向量化结果JSON
   (保留本地)           (保留本地)           (迁移到向量DB)
```

#### 各阶段数据存储位置

| 数据阶段 | 存储位置 | 是否迁移到向量数据库 | 说明 |
|---------|---------|:-------------------:|------|
| **文档加载结果** | 本地 JSON 文件 | ❌ 不迁移 | 存储原始解析内容，与向量检索无关 |
| **文档分块结果** | 本地 JSON 文件 | ❌ 不迁移 | 存储分块文本，是向量化的输入，但本身不需要检索 |
| **向量化结果** | 向量数据库 | ✅ 迁移 | 这才是需要进行相似度检索的数据 |

#### 为什么只迁移向量化结果？

1. **向量数据库的核心能力是向量检索**
   - 只有 embedding 向量才需要 ANN 索引加速
   - 文档加载和分块结果没有向量，不需要向量检索

2. **保持职责分离**
   - 文档加载 → 解析和存储原始内容
   - 文档分块 → 文本处理和切分逻辑
   - 向量检索 → 语义相似度匹配

3. **追溯机制通过 ID 关联**
   ```
   向量数据库中的记录
         │
         │ chunk_id / chunking_result_id
         ▼
   本地分块结果 JSON
         │
         │ document_id
         ▼
   本地文档加载 JSON
   ```

## 2. 核心概念澄清

### 2.1 迁移不是"向量化"

**重要理解**：迁移到向量数据库 ≠ 把所有数据都转成向量

向量数据库是一个**混合存储系统**，支持存储：

```
┌─────────────────────────────────────────────────────────────┐
│                    向量数据库存储结构                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│   │   向量索引    │    │   元数据存储  │    │  原文存储    │  │
│   │  (HNSW/IVF) │    │  (倒排索引)   │    │  (可选)      │  │
│   └──────────────┘    └──────────────┘    └──────────────┘  │
│         │                   │                   │           │
│         ▼                   ▼                   ▼           │
│   ┌──────────────────────────────────────────────────────┐  │
│   │  Record/Point/Document                               │  │
│   │  {                                                   │  │
│   │    "id": "chunk_001",                                │  │
│   │    "vector": [0.123, -0.456, ...],  ← 向量（密集）    │  │
│   │    "payload/metadata": {            ← 元数据（结构化）│  │
│   │      "source": "技术文档.pdf",                        │  │
│   │      "page": 5,                                      │  │
│   │      "chunk_index": 3                                │  │
│   │    },                                                │  │
│   │    "text": "原始文本内容..."        ← 原文（可选）    │  │
│   │  }                                                   │  │
│   └──────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 数据存储方式对照

| 数据类型 | 存储方式 | 用途 |
|---------|---------|------|
| **向量** | 向量索引（HNSW、IVF 等） | 快速相似度检索 |
| **元数据** | 结构化存储（倒排索引） | 精确过滤和追溯 |
| **原文** | 普通文本存储 | 结果展示和引用 |

## 4. 主流向量数据库对比

### 4.1 开源方案

| 数据库 | 特点 | 适用场景 | 部署方式 |
|-------|------|---------|---------|
| **Milvus** | 功能全面，社区活跃 | 大规模生产环境 | 分布式/单机 |
| **Qdrant** | Rust 编写，性能优异 | 中大规模，注重性能 | 单机/分布式 |
| **Weaviate** | GraphQL API，模块化 | 需要复杂查询 | 单机/分布式 |

|-----|------|---------|
| **Pinecone** | 全托管，开箱即用 | 按向量数和查询量 |

| **阿里云向量检索** | 国内云服务 | 按量计费 |

## 5. 迁移实施方案

### 5.1 数据迁移映射

现有 JSON 结构：
```json
{
    "chunk_id": "chunk_001",
    "embedding": [0.123, -0.456, ...],
    "text": "这是分块内容...",
    "text_hash": "abc123...",
    "source": "技术文档.pdf",
    "page": 5,
    "chunk_index": 3,
    "start_char": 1000,
    "end_char": 1500
}
```

迁移到向量数据库（以 Qdrant 为例）：
```python
from qdrant_client import QdrantClient
from qdrant_client.models import PointStruct, VectorParams, Distance

# 初始化客户端
client = QdrantClient(host="localhost", port=6333)

# 创建集合
client.create_collection(
    collection_name="document_chunks",
    vectors_config=VectorParams(
        size=768,  # 向量维度，根据实际 embedding 模型调整
        distance=Distance.COSINE
    )
)

# 插入数据
client.upsert(
    collection_name="document_chunks",
    points=[
        PointStruct(
            id="chunk_001",
            vector=[0.123, -0.456, ...],  # embedding 向量
            payload={                      # 元数据（非向量）
                "text": "这是分块内容...",
                "text_hash": "abc123...",
                "source": "技术文档.pdf",
                "page": 5,
                "chunk_index": 3,
                "start_char": 1000,
                "end_char": 1500,
                "document_id": "doc_123",
                "chunking_result_id": "cr_456"
            }
        )
    ]
)
```

### 5.2 查询示例

```python
# 向量相似度检索 + 元数据过滤
results = client.search(
    collection_name="document_chunks",
    query_vector=[0.1, 0.2, ...],  # 查询向量
    query_filter={
        "must": [
            {"key": "source", "match": {"value": "技术文档.pdf"}},
            {"key": "page", "range": {"gte": 1, "lte": 10}}
        ]
    },
    limit=10
)

# 返回结果包含：向量相似度得分 + 元数据 + 原文
for result in results:
    print(f"Score: {result.score}")
    print(f"Source: {result.payload['source']}, Page: {result.payload['page']}")
    print(f"Text: {result.payload['text']}")
```

### 5.3 迁移脚本框架

```python
import json
from pathlib import Path
from qdrant_client import QdrantClient
from qdrant_client.models import PointStruct

def migrate_json_to_qdrant(json_dir: str, collection_name: str):
    """
    将 JSON 文件中的向量数据迁移到 Qdrant
    """
    client = QdrantClient(host="localhost", port=6333)
    
    # 遍历所有 JSON 文件
    json_path = Path(json_dir)
    for json_file in json_path.glob("*.json"):
        with open(json_file, 'r', encoding='utf-8') as f:
            data = json.load(f)
        
        points = []
        for chunk in data.get("chunks", []):
            point = PointStruct(
                id=chunk["chunk_id"],
                vector=chunk["embedding"],
                payload={
                    "text": chunk.get("text", ""),
                    "text_hash": chunk.get("text_hash", ""),
                    "source": chunk.get("source", ""),
                    "page": chunk.get("page", 0),
                    "chunk_index": chunk.get("chunk_index", 0),
                    "start_char": chunk.get("start_char", 0),
                    "end_char": chunk.get("end_char", 0),
                    # 保留追溯信息
                    "document_id": chunk.get("document_id", ""),
                    "chunking_result_id": chunk.get("chunking_result_id", "")
                }
            )
            points.append(point)
        
        # 批量插入
        if points:
            client.upsert(
                collection_name=collection_name,
                points=points
            )
            print(f"Migrated {len(points)} chunks from {json_file.name}")

# 使用示例
# migrate_json_to_qdrant("./data/embeddings", "document_chunks")
```

## 6. 迁移注意事项

### 6.1 数据一致性

- [ ] 迁移前备份所有 JSON 文件
- [ ] 迁移后验证数据完整性（向量数量、元数据字段）
- [ ] 保留原有 SQLite 数据库的任务元信息

### 6.2 性能优化

- [ ] 根据数据规模选择合适的索引类型（HNSW、IVF 等）
- [ ] 配置合适的向量维度和距离度量方式
- [ ] 考虑分片策略（针对超大规模数据）

### 6.3 兼容性保证

- [ ] 保持 API 接口兼容，对上层应用透明
- [ ] 保留 `chunk_id`、`chunking_result_id` 等追溯字段
- [ ] 支持回退方案（可切换回 JSON 文件存储）

## 7. 推荐方案

### 7.1 小规模场景（<10 万向量）

**推荐**：保持现有 JSON 文件方案，或使用 **Chroma**（嵌入式）

优点：部署简单，无额外运维成本

### 7.2 中等规模场景（10 万 - 1000 万向量）

**推荐**：**Qdrant** 或 **Milvus Lite**

优点：性能优异，部署灵活，社区支持好

### 7.3 大规模场景（>1000 万向量）

**推荐**：**Milvus 分布式** 或 **云服务**（Pinecone、Zilliz Cloud）

优点：支持横向扩展，高可用性
