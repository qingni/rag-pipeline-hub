# RAG Pipeline Hub 模块链路说明

本文档用于为项目架构图、流程图和可视化说明提供素材。

写作目标：
- 不展开具体代码实现细节
- 重点说明各模块的职责边界、输入输出、上下游关系和核心亮点
- 适合作为后续使用 Figma、draw.io、Excalidraw 等工具绘图的依据

---

## 1. 项目整体定位

`RAG Pipeline Hub` 是一套面向真实业务流程的端到端 RAG 平台，覆盖：

1. 文档加载
2. 文档分块
3. 文档向量化
4. 向量索引
5. 搜索查询
6. 文本生成

与普通 RAG Demo 相比，本项目的特点不是只打通一条最短链路，而是把每个阶段都做成了可独立配置、可调试、可复用、可查看结果的模块化工作台。

---

## 2. 建议的可视化视角

如果要做整体可视化，建议按以下 3 个层面组织：

### 2.1 主处理链

`文档上传/接入 -> 文档解析 -> 文档分块 -> 向量化 -> 索引入库 -> 搜索检索 -> 回答生成`

### 2.2 横向支撑层

- 前端工作台
- FastAPI API 编排层
- Service 业务层
- 外部模型与解析服务
- Milvus 向量库
- 结果存储与历史记录
- 异步任务与进度流
- 智能推荐与查询增强

### 2.3 核心亮点层

- Docling Serve + fallback loaders
- 智能分块策略与参数推荐
- Embedding 模型智能推荐
- 逻辑/物理 Collection 分层
- 混合检索 + Query Enhancement + Reranker
- 流式生成 + 来源引用 + 历史记录

## 3. 端到端链路总览

### 3.1 内容处理链

1. 用户上传原始文档
2. 文档加载模块将不同格式文件解析为统一结果
3. 文档分块模块将长文切分为可检索块
4. 文档向量化模块把块转换为向量
5. 向量索引模块把向量写入 Milvus 并维护 Collection / Index

### 3.2 检索生成链

1. 用户发起查询
2. 搜索查询模块执行查询增强、混合召回、融合与精排
3. 文本生成模块基于检索上下文构造 Prompt
4. LLM 返回最终回答
5. 前端展示回答、引用来源和历史记录

### 3.3 核心数据资产

- 原始上传文件
- 标准化文档结果
- 分块结果
- 向量化结果
- 索引与 Collection 元数据
- 搜索结果与历史
- 生成结果与引用来源

## 4. 模块一：文档加载

### 4.1 模块定位

文档加载模块负责把多种异构文件格式转换为统一的标准化文档结果，是整条 RAG 内容链的入口。

它解决的问题包括：

- 文档格式多样，解析方式不统一
- 高质量解析与稳定性之间需要权衡
- 大文件处理耗时较长
- 表格、图片、公式等结构化信息容易丢失

### 4.2 核心能力

- 支持 20+ 文档格式接入
- 以 `Docling Serve` 为高质量主解析器
- 提供 fallback loaders，保证解析失败时可降级
- 支持异步加载任务与任务状态查询
- 支持文档上传、预览、删除、结果查询
- 输出统一的标准化文档内容结构，供下游模块复用

### 4.3 输入 / 输出

**输入**
- 原始文档文件
- 可选加载器类型
- 可选异步任务参数

**输出**
- 标准化文档结果
- 文档全文文本
- 页级信息
- 表格 / 图片 / 公式等结构化元素
- 处理统计信息

### 4.4 关键亮点

- 不只是“文件上传”，而是完整的文档标准化入口
- 高质量解析与稳定性并重：主解析器 + 降级解析
- 结果持久化，方便后续分块与向量化直接消费
- 支持异步处理，更适合真实业务场景

### 4.5 上下游关系

**上游**
- 用户上传与文档管理动作

**下游**
- 文档分块模块直接消费加载结果

### 4.6 可视化建议

图中可抽象为：

`多格式文档输入 -> 解析与标准化 -> 输出统一文档结果`

建议突出 3 个标签：

- `20+ Formats`
- `Docling + Fallback`
- `Async Tasks`

## 5. 模块二：文档分块

### 5.1 模块定位

文档分块模块负责把长文档转换为可检索、可引用、可追溯的块级内容，是内容处理与语义表示之间的关键桥梁。

### 5.2 核心能力

- 支持按字数、按段落、按标题、语义、父子文档、混合分块
- 支持策略参数配置与预览
- 支持文档特征分析
- 支持智能推荐分块策略与参数
- 支持块结果可视化、历史记录与版本管理
- 支持文本、表格、图片、代码等块类型

### 5.3 输入 / 输出

**输入**
- 标准化文档结果
- 分块策略
- 分块参数

**输出**
- 块列表
- 父块 / 子块关系
- 分块统计信息
- 可供向量化消费的结构化结果

### 5.4 关键亮点

- 不依赖单一 chunk_size 方案，而是支持多策略切换
- 内置智能推荐，能基于文档特征自动推荐策略和参数
- 父子分块、混合分块明显高于普通 RAG demo 的复杂度
- 可预览、可调试、可比较分块结果

### 5.5 上下游关系

**上游**
- 文档加载模块输出的标准化文档内容

**下游**
- 文档向量化模块消费分块结果

### 5.6 可视化建议

图中可抽象为：

`标准化文档 -> 策略选择 / 智能推荐 -> 输出块结果`

建议突出 4 个标签：

- `Multi Strategy`
- `Smart Recommendation`
- `Preview & Debug`
- `Parent-Child / Hybrid`

---

## 6. 模块三：文档向量化

### 6.1 模块定位

文档向量化模块负责把分块后的内容转换为向量表示，为索引和检索提供语义基础。

### 6.2 核心能力

- 支持多种 Embedding 模型
- 支持文本模型与多模态模型
- 支持按文档、按分块结果、按批次进行向量化
- 支持模型智能推荐
- 支持批量处理、重试、缓存、进度跟踪
- 支持结果持久化与历史查看

### 6.3 输入 / 输出

**输入**
- 分块结果
- Embedding 模型
- 向量化参数

**输出**
- 向量列表
- 处理统计
- 向量化结果记录
- 可供索引模块消费的向量资产

### 6.4 关键亮点

- 不只是调用一个 embedding API，而是完整的向量化管理模块
- 支持基于语言、领域、多模态特征做模型推荐
- 支持多模态 Embedding
- 支持批量处理和进度反馈，更接近生产级链路
- 支持结果存储与后续复用

### 6.5 上下游关系

**上游**
- 文档分块模块输出的块数据

**下游**
- 向量索引模块消费向量结果进行入库与索引构建

### 6.6 可视化建议

图中可抽象为：

`块结果 -> 模型选择 / 模型推荐 -> 向量结果`

建议突出 4 个标签：

- `bge-m3 / qwen3 / qwen3-vl`
- `Model Recommendation`
- `Batch & Progress`
- `Cache & History`

---

## 7. 模块四：向量索引

### 7.1 模块定位

向量索引模块负责将向量结果组织、写入、管理并落到 Milvus 中，为搜索查询提供底层检索基础设施。

### 7.2 核心能力

- 基于向量结果创建索引
- 管理 Collection 与 Index
- 支持向量写入、删除、统计、恢复
- 支持逻辑 Collection 与物理 Collection 分层
- 支持稀疏向量与 BM25 支撑混合检索
- 支持索引推荐与管理面板

### 7.3 输入 / 输出

**输入**
- 向量化结果
- Collection / Index 配置
- 检索参数

**输出**
- Milvus Collection
- Index 元数据
- 写入结果与统计信息
- 混合检索基础能力

### 7.4 关键亮点

- 不只是“把向量塞进库里”，而是完整的索引管理模块
- 使用逻辑/物理 Collection 分层，便于多模型、多知识库管理
- 为混合检索预留 sparse 字段和相关数据结构
- 支持统计、恢复和推荐能力，更适合工程化场景

### 7.5 上下游关系

**上游**
- 文档向量化模块输出的向量结果

**下游**
- 搜索查询模块基于索引和 Collection 做检索

### 7.6 可视化建议

图中可抽象为：

`向量结果 -> Collection / Index 管理 -> Milvus`

建议突出 4 个标签：

- `Milvus`
- `Collection Management`
- `Index Statistics`
- `Logical / Physical Split`

## 8. 模块五：搜索查询

### 8.1 模块定位

搜索查询模块负责把用户查询转换为高质量检索结果，是连接“索引基础设施”和“答案生成”的核心编排层。

### 8.2 核心能力

- 支持普通搜索与混合检索
- 支持 Dense + Sparse 双路召回
- 支持 `RRF` 融合
- 支持 `Reranker` 精排
- 支持 Query Enhancement
- 支持多 Collection 联合搜索
- 支持历史记录与结果调试

### 8.3 输入 / 输出

**输入**
- 用户查询
- 检索范围（Collection / Index）
- 检索配置参数

**输出**
- 检索结果列表
- 排序分数与来源信息
- 调试信息
- 供文本生成模块消费的上下文结果

### 8.4 关键亮点

- 明显区别于普通 demo 的纯向量 topK 检索
- 把查询增强、混合检索、融合与精排做成完整链路
- 支持多 Collection 联合检索
- 强调真实检索场景中的稳定性和结果质量治理

### 8.5 上下游关系

**上游**
- 向量索引模块提供 Collection、向量和稀疏检索基础

**下游**
- 文本生成模块消费其返回的上下文片段

### 8.6 可视化建议

图中可抽象为：

`用户查询 -> 查询增强 -> 混合召回 -> 融合 / 精排 -> 上下文结果`

建议突出 5 个标签：

- `Query Enhancement`
- `Dense + Sparse`
- `RRF`
- `Reranker`
- `Result Debugging`

## 9. 模块六：文本生成

### 9.1 模块定位

文本生成模块负责基于检索到的上下文和用户问题，构造 Prompt 并调用 LLM 输出最终回答，是整套系统面向用户的最终交付层。

### 9.2 核心能力

- 支持同步生成与流式生成
- 支持 Prompt 配置与上下文注入
- 支持来源引用
- 支持生成历史记录
- 支持取消生成
- 支持 Markdown 安全渲染

### 9.3 输入 / 输出

**输入**
- 用户问题
- 搜索查询模块返回的上下文片段
- 生成模型与参数

**输出**
- 最终回答
- 流式返回内容
- 来源引用
- 历史记录

### 9.4 关键亮点

- 不是裸调用 LLM，而是建立在检索上下文之上的生成模块
- 支持流式输出，交互体验更接近真实产品
- 结果中保留来源引用，适合知识库问答场景
- 具备历史记录与可回看能力

### 9.5 上下游关系

**上游**
- 搜索查询模块提供上下文结果

**下游**
- 面向最终用户输出答案与来源

### 9.6 可视化建议

图中可抽象为：

`上下文结果 + 用户问题 -> Prompt 组装 -> LLM 生成 -> 回答与引用`

建议突出 4 个标签：

- `Sync / Streaming`
- `Prompt & Context`
- `Source Citation`
- `History`

## 10. 横向支撑能力

这一部分不属于某一个单独模块，但在架构图中非常值得作为“支撑层”出现。

### 10.1 前端工作台

- 基于 `Vue 3 + Pinia + Vue Router + TDesign`
- 每个核心阶段都有独立页面
- 强调可配置、可调试、可查看结果

### 10.2 后端 API 与业务编排

- 基于 `FastAPI`
- 采用 API 路由层、Service 业务层、Provider 适配层的组织方式
- 负责把前端工作台和底层能力串起来

### 10.3 外部服务

- `Docling Serve`
- OpenAI-compatible Embedding 服务
- OpenAI-compatible LLM 服务
- Reranker 服务

### 10.4 数据与结果存储

- 上传文件目录
- 模块结果 JSON 文件
- SQLite / PostgreSQL 元数据存储
- Milvus 向量数据存储

### 10.5 异步与进度能力

- 加载异步任务
- 分块任务队列
- 向量化进度流
- 流式生成返回

### 10.6 智能化能力

- 分块策略与参数推荐
- Embedding 模型推荐
- 查询增强
- Contextual Retrieval 风格增强
- 检索结果质量控制

### 10.7 工程化能力

- 历史记录
- 结果持久化
- fallback 策略
- 健康检查
- 可调试和可回溯
